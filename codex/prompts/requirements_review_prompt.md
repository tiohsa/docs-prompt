# レビュー観点

1. **構成・網羅性**

   * 背景・目的 / スコープ / ステークホルダー / 機能要件 / 非機能要件 / 制約 / リスク 等、
     必要な章が一通り含まれているか。
2. **明確性・具体性**

   * あいまいな表現（例：「適宜」「十分に」「できるだけ」等）が多すぎないか。
   * 概念と例が整理されているか。
3. **一貫性・整合性**

   * 章ごとに矛盾する記述がないか。
   * スコープと機能要件、制約条件が整合しているか。
4. **非機能要件の明確さ**

   * 性能・可用性・セキュリティ・運用などの非機能要件が測定可能/検証可能な形で書かれているか。
5. **リスクと前提条件**

   * 重要な前提条件やリスクが明記されているか。
   * 「もし前提が崩れたとき」の影響が考慮されているか。

---

# 出力要件

次の構成で Markdown を出力してください。

## 1. 総評

* 文書全体の印象と、主な長所・懸念点を 3〜5 行でまとめてください。

## 2. 指摘事項一覧

* 表形式で、以下の列を持つテーブルを作成してください。

| No  | 重要度(高/中/低) | 箇所(章・節) | 問題の内容 | なぜ問題か | 改善提案 |
| --- | ---------------- | ------------ | ---------- | ---------- | -------- |

* 主要な指摘のみで構いません（10〜20件程度を目安）。

## 3. 改善後のサンプル抜粋

* 特に改善効果が大きいと思われる 1〜3 箇所について、
  「修正前」と「修正後（提案）」の両方を Markdown で示してください。

## 4. 追加で確認すべき質問リスト

* 関係者に確認すべき具体的な質問を箇条書きで列挙してください。

# 注意

* 元の文書に書かれていない情報を勝手に事実として追加しないでください。
* どうしても推測が必要な場合は、「推測」と明記してください。
